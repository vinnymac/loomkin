# Roadmap: Loomkin — Agent Orchestration Visibility & Control

## Overview

This milestone enhances Loomkin's existing Teams layer to give humans full real-time visibility into
agent-to-agent communication and natural intervention controls. The work proceeds in priority order:
first decompose the 4,714-line monolith and close signal-delivery gaps so the foundation is solid, then
wire live visibility surfaces (comms feed, task graph, crash recovery), then add intervention controls
(chat injection, approval gates, confidence triggers), and finally add dynamic tree spawning safety
and the leader research protocol. Each phase delivers one coherent, testable capability.

## Phase Details

### Phase 1: Monolith Extraction
**Goal**: workspace_live.ex is decomposed into focused LiveComponents so new features can be added without touching a 4,714-line file
**Depends on**: Nothing (first phase)
**Requirements**: FOUN-01
**Success Criteria** (what must be TRUE):
  1. workspace_live.ex acts as a pure orchestrator with no inline rendering logic (reduced from 4,714 to 3,968 lines; remaining orchestration code — signal dispatch, PubSub handlers, agent card management — stays until Phase 2 TeamBroadcaster extraction)
  2. Agent cards, comms feed, team dashboard, inspector panel, and intervention controls each exist as independent LiveComponents with their own state
  3. Each extracted component has its own signal subscriptions and passes only minimal assigns from the parent LiveView
  4. Existing functionality (chat, pause/resume, reply-to-agent, permission gates, ask-user, inspector) works identically after extraction
  5. A LiveView integration test verifies the mission control layout renders all components correctly

### Phase 2: Signal Infrastructure
**Goal**: A TeamBroadcaster GenServer sits between the Signal Bus and LiveView, all signal topics are generated through a Topics module, and dead subscriptions are cleaned up on process termination
**Depends on**: Phase 1
**Requirements**: FOUN-02, FOUN-03
**Success Criteria** (what must be TRUE):
  1. WorkspaceLive subscribes to TeamBroadcaster summary topics rather than raw Jido Signal Bus events directly
  2. TeamBroadcaster debounces high-frequency signals (50ms windows) and batches activity summaries before forwarding to LiveView
  3. All signal topic strings are generated by Loomkin.Teams.Topics module — no raw string interpolation in subscription or publish calls
  4. Agent and team signal bus subscriptions are explicitly unsubscribed in terminate/2 callbacks, with no dead subscriber accumulation across agent lifecycle
  5. With 10 concurrent streaming agents, the WorkspaceLive process message queue length stays under 50 during a sustained run
**Plans**: 5 plans

### Phase 3: Live Comms Feed
**Goal**: Agent-to-agent peer messages appear in the comms feed for all teams including dynamically spawned sub-teams, and newly spawned agents auto-insert into the UI without a page reload
**Depends on**: Phase 2
**Requirements**: VISB-01, VISB-02
**Success Criteria** (what must be TRUE):
  1. When a specialist agent sends a peer message to another agent in the same team, that message appears in the comms feed within one second
  2. When a leader spawns a sub-team, agents from that sub-team appear as new agent cards in the grid without requiring a page reload
  3. Agent-to-agent peer messages from child sub-teams appear in the comms feed alongside messages from the root team
  4. The comms feed uses LiveView stream/3 — adding a new message does not trigger a full list re-render (verifiable via browser DevTools DOM patch size)
  5. Agent color identity is consistent: each agent's messages and card use the same color across all feed updates

### Phase 4: Task Graph & Crash Recovery
**Goal**: The task dependency graph shows blocked-by relationships visually, and OTP agent crashes are reflected in the UI as recovered status without a manual page refresh
**Depends on**: Phase 3
**Requirements**: VISB-03, VISB-04
**Success Criteria** (what must be TRUE):
  1. The task panel displays a visual dependency graph where blocked tasks show which upstream task is blocking them, not just a flat list
  2. When a task transitions from blocked to unblocked (its dependency completes), the UI updates the dependency graph without a reload
  3. When an agent process crashes and OTP restarts it, the agent card changes from an error state to a recovered/running state within two seconds with no user action
  4. The agent card displays an explicit error state (distinct from paused and idle) during the window between crash and restart
  5. A crashed agent that never restarts (e.g., max restart limit hit) remains in error state with an escalation indicator, not silently disappearing

### Phase 5: Chat Injection & State Machines
**Goal**: A human can broadcast a message to the entire team conversation (not just reply-to-agent), and agent pause state is strictly separated from permission-pending state via typed state machines
**Depends on**: Phase 4
**Requirements**: INTV-01, INTV-04
**Success Criteria** (what must be TRUE):
  1. The message composer has a mode where sending broadcasts to the entire team conversation rather than a single agent, and all agents receive the message in their next checkpoint context
  2. Broadcast messages appear in the comms feed labeled as coming from the human operator, not from a specific agent
  3. Pausing an agent that has a pending permission request does not clobber the permission — the permission must be resolved first
  4. Resuming a paused agent requires providing new guidance; approving a permission gate requires a yes/no for the specific tool call — the two UI controls are visually distinct
  5. Agent state correctly transitions through idle, running, paused, permission-pending, and approval-pending without any state being silently overwritten by another

### Phase 6: Approval Gates
**Goal**: Agents can pause at critical junctures and await human sign-off via a checkpoint-based approval gate that is visually and mechanically distinct from the existing permission hook system
**Depends on**: Phase 5
**Requirements**: INTV-02
**Success Criteria** (what must be TRUE):
  1. An agent can call a RequestApproval tool at a critical juncture, which pauses the agent loop and surfaces a high-visibility approval card in the UI
  2. The approval gate UI is visually distinct from the permission hook UI — different card design, different action labels, different positioning
  3. When the human approves, the agent resumes from the exact checkpoint where it paused; when denied, the agent receives the denial as context and continues
  4. If the leader agent is paused at an approval gate, the UI shows a persistent team-wide banner indicating the team is waiting for human sign-off
  5. An approval gate with a timed-out response defaults to deny, not auto-approve, and the agent sees the timeout reason in its resumed context

### Phase 7: Confidence Triggers
**Goal**: Agents automatically surface a question to the human when their confidence drops below a threshold, with rate limiting to prevent interrupt fatigue
**Depends on**: Phase 6
**Requirements**: INTV-03
**Success Criteria** (what must be TRUE):
  1. When an agent determines it is uncertain about a decision, it calls the AskUserQuestion tool and the question surfaces to the human in the existing AskUser UI
  2. Multiple low-confidence questions from the same agent are batched into a single human review card rather than appearing as separate interrupts
  3. An agent cannot trigger more than one confidence-threshold question per five minutes regardless of how many uncertain decisions it encounters
  4. The human can dismiss a confidence-triggered question with "Let the team decide" and the agent proceeds autonomously using collective-decide fallback
  5. The confidence trigger pathway is wired from AgentLoop through the signal bus to AskUserComponent — not bypassing any existing component

### Phase 8: Dynamic Tree Visibility
**Goal**: Nested sub-teams at arbitrary depth auto-appear in the UI via recursive subscription, and the ChildTeamCreated signal is reliably published with leader ownership and proper termination on dissolve
**Depends on**: Phase 7
**Requirements**: TREE-01, TREE-02
**Success Criteria** (what must be TRUE):
  1. When a leader spawns a child sub-team, that sub-team appears as a nested node in the agent tree UI without any user action
  2. When a child sub-team dissolves, its node and all descendant nodes are removed from the tree UI and signal subscriptions are cleaned up
  3. Sub-teams at depth 2 (grandchild teams) also appear in the tree and their agents' peer messages appear in the comms feed
  4. If a leader agent crashes and OTP restarts it, the previously spawned child agents are terminated (not left running as ghosts) before the leader restarts
  5. The agent tree shows each team node with: team name, depth indicator, agent count, and parent relationship — all updated live as the tree changes

### Phase 9: Spawn Safety
**Goal**: Before a leader spawns an expensive sub-tree, a budget check and human approval gate run first — and the gate surfaces the estimated cost so the human can make an informed decision
**Depends on**: Phase 8
**Requirements**: TREE-03
**Success Criteria** (what must be TRUE):
  1. When a leader agent intends to spawn a sub-team, the UI surfaces an approval card showing the planned sub-team composition and estimated cost before any spawning occurs
  2. If the human approves, spawning proceeds; if the human denies, the leader receives the denial as guidance and can propose an alternative approach
  3. If remaining team budget is below the threshold for the planned sub-tree, the spawn is blocked and the leader is notified even without human intervention
  4. The max-agents-per-team and max-nesting-depth limits are enforced server-side and surfaced in the UI as a warning when the leader approaches them
  5. Auto-approve mode (for power users who trust the leader) can be toggled on — in this mode, spawns proceed if within budget without waiting for human input

### Phase 10: Leader Research Protocol
**Goal**: The leader agent follows a multi-step research protocol — spawning research sub-agents, synthesizing their findings, and then posing an informed question to the human — rather than asking humans prematurely
**Depends on**: Phase 9
**Requirements**: LEAD-01, LEAD-02
**Success Criteria** (what must be TRUE):
  1. On receiving a complex task, the leader spawns one or more research sub-agents, waits for their synthesis, and only then surfaces a clarifying question to the human
  2. The research phase is visible in the UI: research sub-agents appear in the tree with a "researching" status, and the leader shows "awaiting synthesis" state
  3. The leader's question to the human includes a summary of what the research sub-agents found, so the human can answer from an informed position
  4. Cost attribution for the research sub-team is visible in the UI before the leader asks the human — the human can see what the research cost before deciding next steps
  5. The leader role config includes the research orchestration prompts and multi-step protocol definition as code-level configuration (not a runtime UI)
